# База знаний команды

Этот раздел — для **людей в команде**. Страницы здесь — каноничная
"база знаний"; они также подаются в RAG / vector database команды,
поэтому тот же контент достижим через in-app поиск и chat-ассистенты.

| Страница | Для | Что найдёте |
|------|-----|------------------|
| [Онбординг нового разработчика](./onboarding.md) | Новые инженеры | План на День 1 / Неделю 1 / Месяц 1, порядок чтения, стартовые тикеты |
| [QA — процесс и база знаний](./qa.md) | QA-команда | Тест-планы, шаблоны багов, severity, регрессионные горячие точки |
| [Product Management — процесс и база знаний](./pm.md) | Продуктовая команда | Шаблоны PRD, RICE, release notes, контекст проекта |

## Как используется этот раздел

- **Онбординг** — менеджеры направляют новичков сюда в день 1.
- **Тест-кейсы** — QA пишет тесты против регрессионных горячих точек
  и пер-модульных ключевых фич, задокументированных под
  [Modules](../modules/overview.md).
- **Новые фичи** — Product пишет PRD со ссылкой на соответствующий
  модуль + flow фичи, затем engineering реализует против
  установленных паттернов.
- **RAG / поиск** — vector DB команды индексирует эти страницы, поэтому
  пользователи могут спрашивать "как работает payment approval" и получать
  правильный пассаж.

## Документация = база знаний = онбординг

Хорошо написанная страница должна служить всем трём аудиториям. Когда
пишете новую страницу, спросите:

1. Будет ли это понятно **новому инженеру** без контекста платформы?
   (Если нет — добавьте Summary-параграф.)
2. Найдёт ли **QA-инженер** регрессионные риски? (Добавьте секцию
   "Failure modes" или "Hot-spots", если уместно.)
3. Будет ли **PM** знать о влиянии фичи на роли, мобильное и
   compliance? (Используйте per-module таблицу Key Features.)
4. Будет ли **vector DB** аккуратно чанкать страницу? (Самодостаточные
   секции, абсолютные ссылки — никаких "see above".)
