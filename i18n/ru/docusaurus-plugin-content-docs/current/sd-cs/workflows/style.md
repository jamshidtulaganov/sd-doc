# Как написать страницу воркфлоу о фиче

Это руководство рассказывает, **как документировать одну фичу** sd-cs
(один отчёт, одну сводную таблицу, одну страницу справочника, один
эндпоинт API), чтобы новый сотрудник мог прочитать её один раз и
уверенно использовать фичу.

Это компаньон по написанию для платформенных
[Стандартов проектирования воркфлоу](/docs/team/workflow-design) — та
страница покрывает *проектирование* воркфлоу; эта страница покрывает
*написание страницы* о существующем воркфлоу.

## Аудитория и цель

Профиль читателя:

- Новый сотрудник в свой **первый месяц**.
- Прочитал [обзор sd-cs](../overview.md) и
  [страницу архитектуры](../architecture.md), поэтому знает, что
  существуют две базы данных (`cs_*` control plane, `d0_*` хранилище)
  и что филиалы скоупят данные дилера.
- **Не** эксперт по Yii.
- Хочет ответить на один из двух вопросов:
  1. *«Что делает эта фича и как мне её использовать?»*
  2. *«Если клиент говорит, что она сломана, где мне искать?»*

Если страница делает оба, вы написали хорошую страницу.

## Что мы не пишем

- **Туториалы** («кликните сюда, потом сюда»), которые устаревают
  каждый релиз.
- **Дампы кода** тел контроллеров — вместо этого ссылайтесь на файл.
- **Спек / дизайн-доки** для нереализованных фич — это территория PRD.
- **Дублирование справочника** — список полей живёт в
  [data schemes](../data-schemes.md). Ссылайтесь, не повторяйте.

## Обязательные секции

Каждая страница о фиче имеет эти восемь секций в этом порядке. **Не
добавляйте новые секции верхнего уровня** — если что-то не подходит,
скорее всего, оно принадлежит другому документу и должно быть
ссылкой.

| # | Секция | Что туда идёт |
|---|--------|---------------|
| 1 | **Назначение** | Один абзац. Бизнес-вопрос, на который отвечает фича. Без кода, без UI. |
| 2 | **Кто использует** | Роли + ключи доступа (`module.controller.action`), которые её защищают |
| 3 | **Где это находится** | URL, путь контроллера, ключевые view-файлы, связанные классы моделей |
| 4 | **Воркфлоу** | Нумерованный список. Путь пользователя от «открыл страницу» до «получил ответ». Mermaid sequence-диаграмма, если 3+ участников |
| 5 | **Правила** | Bullet-список. Валидация, скоупинг, edge-кейсы. Каждое правило начинается с глагола (*«Фильтрует …»*, *«Исключает продукты в …»*) |
| 6 | **Источники данных** | Двухколоночная таблица: *Таблица* / *БД и соединение*. Различайте `cs_*` (control) и `d0_*` / `d0_fN_*` (хранилище) |
| 7 | **Подводные камни** | Две-три вещи, которые удивляют новых сотрудников (медленные запросы, NULL-семантика, несоответствие префикса, …) |
| 8 | **См. также** | 2–4 ссылки — связанная страница архитектуры, соседние фичи, исходный файл |

## Голос и стиль

- **Подлежащее — фича, не пользователь.** Пишите *«Отчёт агрегирует
  продажи по филиалу»*, не *«Вы можете агрегировать продажи по
  филиалу»*.
- **Глаголы в настоящем изъявительном.** *«Загружает», «фильтрует»,
  «пишет»* — не *«будет загружать», «должен фильтровать»*.
- **Одно предложение на буллет.** Если нужна запятая-затем-но,
  разбейте.
- **Числа и собственные имена точные.** *«7 активных филиалов в
  демо-БД»*, не *«около 7 филиалов»*. Запустите запрос, если не
  знаете.
- **Перекрёстные ссылки, не повторение.** Сошлитесь на страницу
  архитектуры один раз для механики multi-DB / multi-filial; не
  объясняйте их заново.
- **Один скриншот — нормально, три — слишком много.** Страницы
  должны оставаться поисковыми и переводимыми; опирайтесь на
  нумерованный список воркфлоу.

## Визуальная таксономия

Если на странице есть диаграмма, используйте платформенно-стандартные
цвета из [Галереи диаграмм](/docs/diagrams):

| Цвет | Класс | Значение |
|------|-------|----------|
| Синий | `action` | Стандартный шаг |
| Янтарный | `approval` | Требует ревью / одобрения |
| Зелёный | `success` | Финальный OK / закрытое состояние |
| Красный | `reject` | Неудача / отменённое состояние |
| Серый | `external` | Внешняя система (1C, Telegram, FCM, …) |
| Фиолетовый | `cron` | Запланированное / по расписанию |

## Секция Правила — планка качества

Секция **Правила** — это место, где живёт большая часть онбординг-ценности.
Хорошее правило **фальсифицируемо**: читатель может проверить его по
коду или данным и сказать, истинно ли оно сегодня. Плохие правила
размытые.

| Плохо | Хорошо |
|-------|--------|
| *«Фильтры работают как ожидается.»* | *«Фильтр `date` применяется к `order.DATE`; фильтр `dateLoad` применяется к `order.DATE_LOAD`. Дефолт — `date`. Другие значения молча приводятся к `date` (whitelist в `AkbController::actionPivotData`).»* |
| *«Только админы видят всех дилеров.»* | *«`getOwnFilials()` возвращает все активные филиалы, когда `Yii::app()->user->isAdmin()` равно true; иначе возвращает пересечение `cs_user_filial` и `d0_filial.active='Y'`.»* |

Если вы не можете написать фальсифицируемое правило, **читайте больше
кода** — правило уже в источнике, вы просто его пока не нашли.

## Секция Воркфлоу — планка качества

Шаг воркфлоу **наблюдаем извне функции**: он либо меняет UI,
отправляет HTTP-запрос, пишет в БД, либо показывает результат. Шаги
реализации («перебирает массив филиалов») принадлежат секциям
**Правила** или **Подводные камни**, не воркфлоу.

```
Плохой шаг воркфлоу:  "Конструирует CDbCriteria с диапазоном дат."
Хороший шаг воркфлоу: "Пользователь выбирает диапазон дат и нажимает Apply."
                      "Страница выполняет POST фильтров на /pivot/akb/pivotData."
                      "Сервер возвращает NDJSON, одна строка на филиал."
                      "Страница рендерит pivot-таблицу на клиенте."
```

## Секция Источники данных — что включить

Для каждой таблицы, на которую ссылается фича, перечислите:

- **Схема** — `cs3_demo` (control) или `b_demo` (хранилище).
- **Семейство префиксов** — `cs_`, `d0_` или `d0_fN_` (по филиалу).
- **Зачем читается** — одна строка.

**Не** документируйте колонки повторно; ссылайтесь на
[data schemes](../data-schemes.md).

## Frontmatter

Каждая страница использует:

```yaml
---
sidebar_position: <integer, unique within workflows/>
title: <module> · <feature> # e.g. "report · Sale"
---
```

Заголовок `<module> · <feature>` — это то, что показывается в боковой
панели и поиске. Держите его коротким и машинно-сортируемым.

## Ревью своей страницы

Перед открытием PR прогоните эти четыре проверки:

1. **Найдите реального нового сотрудника** (или симулируйте). Дайте
   ему страницу. Он должен быть в состоянии оперировать фичей в
   staging-окружении в течение 10 минут.
2. **Каждое правило фальсифицируемо.** Перечитайте секцию **Правила**
   и спросите *«Могу ли я опровергнуть это по коду или БД?»* — если
   нет, перепишите или удалите.
3. **Ни одна секция не пропущена.** Все восемь обязательных секций
   присутствуют по порядку.
4. **Перекрёстные ссылки работают.** Запустите `npm run build`
   локально; предупреждения о сломанных ссылках — это CI failure.
