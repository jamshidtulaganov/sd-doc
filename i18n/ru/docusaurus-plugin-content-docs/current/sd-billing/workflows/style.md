# Как писать страницу feature workflow

Это руководство говорит, **как документировать одну фичу** sd-billing (одна
интеграция шлюза, один админ-экран, один API endpoint, одна cron-
команда), чтобы новый сотрудник прочитал его один раз и уверенно
использовал фичу.

Это writing-companion к платформенным
[стандартам дизайна workflow](/docs/team/workflow-design) — та страница
покрывает *дизайн* workflow; эта покрывает *написание страницы*
о существующем workflow.

## Аудитория и цель

Профиль читателя:

- Новый сотрудник в **первый месяц**.
- Прочитал [обзор sd-billing](../overview.md) и
  [авторизация и доступ](../auth-and-access.md), так что знает, что система
  работает на **одной billing-БД** (префикс `d0_*`) с per-tenant
  скопом лицензии/подписки, и что `Access::check()` гейтит каждый
  экран.
- **Не** Yii-эксперт.
- Хочет ответить на один из двух вопросов:
  1. *«Что эта фича делает и как мне её использовать?»*
  2. *«Если клиент скажет, что сломалось, где смотреть?»*

Если страница делает оба — вы написали хорошую страницу.

## Что мы не пишем

- **Туториалы** («кликни сюда, потом сюда»), которые устаревают каждый релиз.
- **Code dumps** тел контроллеров — лучше ссылка на файл.
- **Spec / design docs** для непостроенных фич — это территория PRD.
- **Дублирование справочников** — список таблиц живёт в
  [data scheme](../data-scheme.md). Ссылайтесь, не повторяйте.

## Обязательные секции

Каждая страница фичи имеет эти восемь секций в этом порядке. **Не
добавляйте новые top-level секции** — если что-то не помещается, вероятно,
оно принадлежит другому документу и должно быть связано ссылкой.

| # | Секция | Что туда идёт |
|---|---------|-----------------|
| 1 | **Назначение** | Один абзац. Бизнес-вопрос, на который отвечает эта фича. Без кода, без UI. |
| 2 | **Кто использует** | Роли + ключи доступа (`module.controller.action`), которые её гейтят |
| 3 | **Где живёт** | URL, путь контроллера, ключевые view-файлы, связанные классы моделей |
| 4 | **Workflow** | Нумерованный список. Путь пользователя от «открыть страницу» до «получить ответ». Mermaid sequence-диаграмма, если 3+ актора |
| 5 | **Правила** | Bullet-список. Валидация, scoping, edge-cases. Каждое правило начинается с глагола (*"Фильтрует по …"*, *"Отклоняет если …"*) |
| 6 | **Источники данных** | Таблица из двух колонок: *Таблица* / *Назначение*. Все таблицы живут в одной billing-БД (префикс `d0_*`); отметьте tenant-скоп (по `DILER_ID`, `DISTR_ID` или country) где уместно |
| 7 | **Подводные камни** | Две-три вещи, которые удивляют новых сотрудников (gateway sign verification, NULL-семантика, захардкоженные токены, …) |
| 8 | **См. также** | 2–4 ссылки — связанная страница архитектуры, соседние фичи, исходный файл |

## Голос и стиль

- **Субъект — фича, не пользователь.** Пишите *«Endpoint проверяет
  подпись Click перед записью транзакции»*, не *«Вы можете
  проверить подпись Click»*.
- **Глаголы в настоящем индикативе.** *«Загружает», «отклоняет», «пишет»* —
  не *«будет загружать», «должен отклонять»*.
- **Одно предложение на bullet.** Если нужна запятая-однако-но, разделите.
- **Числа и собственные имена точные.** *«3 активных шлюза в
  demo-тенанте»*, не *«несколько шлюзов»*. Запустите запрос, если не
  знаете.
- **Кросс-ссылка, не повторение.** Сослаться на overview один раз для
  механики single-DB / per-tenant; не объяснять снова.
- **Один скриншот — нормально, три — слишком много.** Страницы должны оставаться
  searchable и translatable; полагайтесь на нумерованный список workflow.

## Визуальная таксономия

Если на странице есть диаграмма, используйте платформенно-стандартные цвета из
[Diagram gallery](/docs/diagrams):

| Цвет | Класс | Значение |
|--------|-------|---------|
| Blue | `action` | Стандартный шаг |
| Amber | `approval` | Требует review / approval |
| Green | `success` | Финальный OK / закрытое состояние |
| Red | `reject` | Сбой / cancelled-состояние |
| Grey | `external` | Внешняя система (Click, Payme, Paynet, Telegram, SMS, 1C, …) |
| Purple | `cron` | Scheduled / time-driven |

## Секция Правила — планка качества

Секция **Правила** — где находится большая часть онбординг-ценности. Хорошее
правило **фальсифицируемо**: читатель может проверить его по коду или
данным и сказать, верно ли оно сегодня. Плохие правила расплывчаты.

| Плохо | Хорошо |
|-----|------|
| *«Проверка подписи работает как ожидается.»* | *«`ClickTransaction::checkSign()` сравнивает `sign_string` с `md5(service_id + amount + transaction_id + SECRET_KEY)`; возвращает `false` на несовпадении, что заставляет endpoint отвечать ошибкой `-1`.»* |
| *«Только админы видят всех дилеров.»* | *«`Access::check('operation.dealer.payment', Access::SHOW)` бросает `CHttpException(403)` для любого пользователя, у которого нет бита; `IS_ADMIN` и `ROLE_ADMIN` короткозамыкают на allow через `Access::has()`.»* |

Если не можете написать фальсифицируемое правило, **читайте больше кода** —
правило уже в исходниках, вы просто его ещё не нашли.

## Секция Workflow — планка качества

Шаг workflow **наблюдаем извне функции**: он
либо изменяет UI, отправляет HTTP-запрос, пишет в БД, либо показывает
результат. Имплементационные шаги («строит SQL-условие») принадлежат
**Правилам** или **Подводным камням**, не workflow.

```
Bad workflow step:  "Constructs QueryBuilder condition for date range."
Good workflow step: "User picks a date range and clicks Apply."
                    "Page POSTs filters to /report/revenue/getData."
                    "Server returns JSON, one row per dealer."
                    "Page renders the revenue grid client-side."
```

## Секция Источники данных — что включать

Для каждой таблицы, на которую ссылается фича, перечислите:

- **Таблица** — `d0_<name>` (все таблицы делят одну billing-БД).
- **Tenant-скоп** — как запрос ограничивается одним тенантом
  (например, `DILER_ID`, `DISTR_ID`, `COUNTRY_ID`).
- **Зачем читается** — одна строка.

**Не** редокументируйте колонки; вместо этого ссылайтесь на
[data scheme](../data-scheme.md).

## Frontmatter

Каждая страница использует:

```yaml
---
sidebar_position: <integer, unique within workflows/>
title: <module> · <feature> # e.g. "api · Click gateway"
---
```

Заголовок `<module> · <feature>` — то, что показывается в сайдбаре и
поиске. Держите его коротким и machine-sortable.

## Ревью своей страницы

Перед открытием PR прогоните эти четыре проверки:

1. **Найдите реального нового сотрудника** (или симулируйте). Дайте ему страницу.
   Он должен быть способен оперировать фичей в staging-окружении за
   10 минут.
2. **Каждое правило фальсифицируемо.** Перечитайте секцию **Правила** и
   спросите *«Мог бы я доказать это неверным по коду или БД?»* — если нет,
   перепишите или удалите.
3. **Ни одна секция не пропущена.** Все восемь обязательных секций присутствуют
   в порядке.
4. **Кросс-ссылки работают.** Запустите `npm run build` локально; broken-link
   warnings — это сбои CI.
