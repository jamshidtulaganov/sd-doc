# Кеширование

## Три Redis-базы, у каждой своя задача

| Redis DB | Компонент | Назначение |
|----------|-----------|---------|
| **db0** | `redis_session` (`CRedisCache`) | Хранилище HTTP-сессий. Префикс: `HTTP_HOST:` |
| **db1** | `queueRedis` (`RedisConnection`) | Очередь фоновых задач (payload-ы `BaseJob`) |
| **db2** | `redis_app` (`CRedisCache`) | Кеш приложения, скоупированный через `TenantContext` |

`FLUSHDB` на одной базе **не** влияет на остальные.

## Когда использовать кеш

- ✅ Lookup-ы, которые читаются в 100× чаще, чем пишутся (типы цен,
  дерево категорий, иерархия authitem, список филиалов).
- ✅ Дорогие агрегаты (KPI дашборда, итоги по дебиторке).
- ❌ Per-request мемоизация — используйте статическое свойство.
- ❌ Всё, что должно быть транзакционно консистентным — храните в MySQL.

## TTL — значения по умолчанию

| Данные | TTL | Обоснование |
|------|-----|-----------|
| Auth items / RBAC | 600s | Дёшево пересчитать, устаревшая авторизация рискована |
| Список филиалов | 3600s | Меняется очень редко |
| Дерево категорий каталога | 1800s | Меняется несколько раз в день |
| KPI дашборда | 60s | Snappy UI, near-real-time |
| Отчёты (тяжёлый SQL) | 300s | Уменьшает нагрузку на БД |

## Инвалидация

Всегда инвалидируйте в момент записи. Паттерны:

```php
// After saving a price
$cache = Yii::app()->tenantContext->scopedFilial();
$cache->delete('catalog:price-types');
$cache->delete("product:{$id}:prices");
```

Для cache *bumps* (versioned ключей):

```php
$ver = $cache->increment('catalog:version');
$key = "catalog:tree:v{$ver}";
```

Это избегает fan-out инвалидации, когда много ключей зависят от одного источника.
