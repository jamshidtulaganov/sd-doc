# Аутентификация и роли

:::note Scope
Эта страница документирует роли в **sd-main** (CRM дилера). Числовые ID ролей здесь — это НЕ те же ID, что в [sd-billing](../sd-billing/auth-and-access.md), даже если значения совпадают. Когда страница модуля sd-main говорит "Approve: 1, 2, 9", смотрите таблицу на этой странице.
:::

## Роли (канонические)

Определены в `protected/config/auth.php`:

| ID | Роль | Примечания |
|----|------|-------|
| 1 | Super Administrator | Все филиалы, все возможности |
| 2 | Administrator Filial | Один филиал, полный набор фич |
| 3 | Diler | Дилер, ограниченный администратор |
| 4 | Agent | Мобильный торговый агент |
| 5 | Operator | Бэк-офис оператор |
| 6 | Kassir | Кассир |
| 7 | Partner | Внешний партнёрский доступ |
| 8 | Supervayzer | Супервайзер / тимлид |
| 9 | Manager | Менеджер по продажам |
| 10 | Expeditor | Экспедитор |
| guest | Guest | Анонимный |

Иерархия частичная: `1 → 2 → 3 → 4 → guest`. Роли 5–10 наследуют только
от `guest` и опираются на пер-permission гранты в `authassignment`.

## Каналы аутентификации

| Поверхность | Механизм |
|---------|-----------|
| Web admin | Куки сессии + `WebUser` |
| Мобильное приложение | Bearer-токен через `LoginController` (api3) |
| Online portal / WebApp | Bearer-токен через api4 + Telegram-signed init |
| Server-to-server | Общий секрет на интеграцию |

## Парольная политика

- Минимальная длина **8** при создании / смене.
- Хеширование (на аккаунт): **только MD5** — и `UserIdentity::authenticate`,
  и `User::validatePassword` сравнивают `md5($plain)` со столбцом
  `PASSWORD`. Миграция на `password_hash()` (bcrypt) запланирована.
- Списка распространённых паролей нет; это запланированное улучшение.

## Обработка отказов

- 5 подряд неуспешных логинов на пользователя → блокировка на 1 минуту.
- 30 неуспешных логинов с одного IP за 10 минут → throttling IP на gateway.
