# Изоляция данных между тенантами

Три слоя складываются, чтобы держать тенантов раздельно:

1. **Сеть** — каждый поддомен — свой vhost; CORS закрыт для
   кросс-поддоменных вызовов.
2. **База данных** — DB-per-tenant. `TenantContext` выбирает БД на каждом
   запросе; fall-through нет.
3. **Кеш** — каждый ключ в `redis_app` префиксован `t:{db}:` (и
   `f:{filial}:` для filial-scoped данных), поэтому коллизии ключей
   между тенантами структурно невозможны.

## Модель угроз

Реалистичные атаки, от которых мы защищаемся:

- **Скомпрометированные креды тенанта** → видит только свою БД.
- **Скомпрометированный app-процесс** → если PHP RCE, атакующий мог бы
  читать всех тенантов. Защита: WAF + container sandbox + минимально
  смонтированные секреты.
- **Устаревший кеш после удаления** → максимум 600 с устаревшего RBAC; заказы
  и платежи никогда не кешируются.

## Антипаттерны

- ❌ `Yii::app()->cache->get(...)` напрямую — обходит префикс тенанта.
- ❌ Жёстко закодированный `connectionString` со ссылкой на дефолтную БД.
- ❌ Cron-задача в цикле по всем тенантам, но пишущая в один общий ключ
  кеша.
