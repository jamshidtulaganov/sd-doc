# ADR 0002 — Мультитенантность: DB-per-customer

- Status: **Accepted**
- Date: 2023-04-02
- Deciders: Platform team

## Контекст

Тенанты сильно различаются по размеру (от 50 до 50 000 заказов / день).
Некоторым требуется data residency. Бэкапы должны быть изолированы по
тенантам. Мы должны поддерживать запросы "отдайте мне мои данные" в
конечное число кликов.

Рассматривались опции:

1. Schema-per-tenant в одной MySQL-БД.
2. **DB-per-customer.**
3. Одна общая схема с колонками `tenant_id`.
4. Per-tenant Mongo-коллекция (отклонено — на команде нет document store).

## Решение

**DB-per-customer.** Одна MySQL-БД на тенанта; поддомен выбирает
БД во время запроса.

## Последствия

- ✅ Per-tenant backup/restore тривиален.
- ✅ Утечки данных между тенантами структурно невозможны на уровне БД.
- ✅ Per-tenant тюнинг (индексы, партицирование).
- ❌ Миграции должны fan-out по всем БД тенантов.
- ❌ Cross-tenant агрегатная отчётность требует отдельного аналитического
  пайпа.
- ❌ Churn пула соединений на app-уровне — решается через persistent-соединения.
